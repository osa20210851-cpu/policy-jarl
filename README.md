# 🛡️ Policy-Jarl: A Policy-Driven RBAC Guard for AI DB Agents
**A Zero-Trust AuthN/AuthZ Guardrail Framework Securing Autonomous AI DB Agents**

Policy-Jarl is an academic attempt at a security middleware architecture designed to bridge the "governance gap" in autonomous agentic workflows. As AI agents transition from simple text generation to executing complex multi-step reasoning plans (such as the ReAct loop) with direct access to database infrastructure, they introduce significant security, compliance, and deterministic control risks.

This framework translates probabilistic AI actions into deterministic, rule-based security decisions by decoupling detection, decision-making, and enforcement.

---

## 📖 Architecture & Separation of Concerns

Policy-Jarl implements a strict **Zero-Trust Architecture** divided into three distinct operational layers:

</p>

1.  **Detection Layer (FastAPI Interceptor Proxy):**
    Intercepts natural language inputs, intent signals, and SQL queries generated by the AI agent. It acts as a translator, mapping probabilistic model outputs into structured JSON payloads for policy evaluation.
2.  **Decision-Making Point (Open Policy Agent - OPA):**
    Evaluates incoming payloads against declarative Policy-as-Code (PaC) rules written in **Rego** (`policies/jarl.rego`). OPA makes independent, deterministic validation verdicts: `ALLOW`, `DENY`, `MASK`, or `ESCALATE`.
3.  **Enforcement Layer (Database Gateway - `db_gate.py`):**
    Functions as the *Credential Sequestration Point*. This is the only microservice containing the actual PostgreSQL database credentials. It acts on OPA's verdict—blocking denied calls, dynamically patching query filters, applying column-level data masking, or holding execution for human approval.

## 🛠️ The Technology Stack
*   **Policy Engine:** [Open Policy Agent (OPA)](https://www.openpolicyagent.org/) using Rego for sub-millisecond, decoupled authorization logic.
*   **Identity Provider (IdP):** [Telegram Bot API](https://core.telegram.org/bots) providing a secure, device-linked, and session-aware identity boundary.
*   **Context Memory:** [Redis](https://redis.io/) serving as a stateful, high-performance database for active user profiles, security counters (to mitigate repeat-offender attacks), and HITL states.
*   **Forensics & Observability:** [ELK Stack](https://www.elastic.co/what-is/elk-stack) (Elasticsearch, Logstash, Kibana) for real-time audit logging and visual security operations center (SOC) dashboards.
*   **AI Engine:** [HKUDS Nanobot](https://github.com/HKUDS/nanobot) worker powered by the [Google Gemini 3.1](https://ai.google.dev/gemini-api) model.
*   **Database:** [PostgreSQL](https://www.postgresql.org/) hosting production-grade business schemas.

## 🛡️ OPA Security Constitution (`policies/jarl.rego`)

Security validation policies are processed in strict priority order to defend against OWASP Top 10 LLM vulnerabilities:
*   **Layer 1: Global Safety Lock (System Kill-Switch):** Immediate deny if system state is set to `LOCKED` (Incident Response).
*   **Layer 2: Semantic Mismatch Detection:** Prevents prompt/intent manipulation by blocking actions where the agent's stated intent (e.g., "read-only lookup") contradicts the generated SQL command (e.g., `DELETE`, `DROP`, `ALTER`).
*   **Layer 3: Sovereign Override:** Grants full bypass to verified cryptographic `admin` credentials.
*   **Layer 4: Governed Metadata Discovery:** Dynamically rewrites database catalog/schema queries to only expose schemas the active user role is authorized to see.
*   **Layer 5: RBAC Table & Verb Validation:** Enforces strict role-based access limits (e.g., restricts `guest`, `data_clerk`, `hr_analytics`, `ops_manager`, `finance_analyst` to specific tables and SQL verbs).
*   **Layer 6: Dynamic Data Masking:** Intercepts `SELECT` output data and redacts sensitive columns (e.g., PII, salary, or comments fields) on-the-fly for unauthorized roles.
*   **Layer 7: Data Leakage Prevention (DLP):** Blocks outbound model responses if sensitive data or mission drift is detected in the plain-text reply.

## 🧪 Testing & Evaluation Gauntlets

Policy-Jarl includes extensive validation suites to measure guardrail resistance:
*   **`final_exam.py`:** Runs a comprehensive evaluation suite compiling both industrial and adversarial test sets.
*   **`abyss_auditor.py`:** A dedicated adversarial test runner testing resistance to SQL injection, prompt injection, and privilege escalation payloads using `abyss_gauntlet.json`.
*   **`simulate_traffic.py`:** Simulates live multi-role traffic to test logging pipelines and populate Kibana dashboards.

## 💻 How to Run

### 0. Secret Initilization
Fill up a .venv file to include all of the required secrets and API keys to be used by the Docker network + an API key for NANO bot in it's config file.
Look at the `env.example` for a reference. 

### 1. Startup Infrastructure
Launch the Docker Compose environment:
```bash
docker-compose up --build
```
This initializes the FastAPI Officer proxy, OPA engine, database gateways, PostgreSQL range database, Redis context cache, and the ELK logging stack.

### 2. Run the Compliance & Security Scorecard
Install the python requirements locally and verify the system's resistance score:
```bash
# Set up a local python environment
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

# Run the test suite
python3 final_exam.py
```
This runs over 1,000+ test scenarios verifying OPA policy enforcement, RBAC enforcement, and adversarial resistance.
